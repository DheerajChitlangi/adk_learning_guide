📚 What is Google ADK?

🎯 The Story: From API Calls to Intelligent Agents

Imagine you're building a customer service application. Initially, you might think: "I'll just call an LLM API with a prompt." But soon you realize your assistant needs to:

  • Check order status in your database
  • Search knowledge base articles
  • Remember previous conversations
  • Escalate complex issues to human agents
  • Work with different specialized sub-agents

This is where Google's Agent Development Kit (ADK) transforms your journey from simple API calls to building production-ready intelligent agents.

✨ What Makes ADK Special?

Google's ADK is an open-source, code-first framework that applies software development principles to AI agent creation. It's the same framework powering agents within Google products like Agentspace and Google Customer Engagement Suite.

⚡

Code-First Development

Define agent logic, tools, and orchestration directly in Python for ultimate flexibility, testability, and versioning.

🧩

Modular Architecture

Build scalable applications by composing multiple specialized agents into flexible hierarchies.

🛠️

Rich Tool Ecosystem

Utilize pre-built tools, custom functions, OpenAPI specs, MCP tools, or integrate existing tools from LangChain and CrewAI.

🌐

Model Agnostic

While optimized for Gemini, ADK supports various LLMs including OpenAI, Anthropic, and local models via LiteLLM.

☁️

Deploy Anywhere

Easily containerize and deploy on Cloud Run, or scale seamlessly with Vertex AI Agent Engine Runtime.

🎬

Native Multimodal

Built-in support for voice, video, images, and real-time streaming with Gemini's multimodal capabilities.

💡 Key Insight: ADK covers the entire agent lifecycle—from development and testing to evaluation and production deployment. It's not just a library; it's a complete framework for building enterprise-grade agentic systems.

🚀 Getting Started with ADK

📖 Your First Agent: A Personal Assistant

Let's build your first agent step by step. We'll create a simple personal assistant that can answer questions and tell you the current time.

Installation & Setup

  1. Install ADK
    pip install google-adk

    ADK is available for Python, Go, and Java. The release cadence is roughly bi-weekly.

  2. Create Your First Project
    adk create --type=code my_first_agent cd my_first_agent

    This creates a project structure with agent/ directory containing your agent logic.

  3. Configure API Access

    Create a .env file with your credentials:

    Google AI Studio (Fastest)
    # Recommended for prototyping GOOGLE_GENAI_USE_VERTEXAI=0 GOOGLE_API_KEY="YOUR_API_KEY_HERE" # Get your API key from: https://aistudio.google.com/apikey
    ⚠️ Security First: Never hardcode credentials in your code! Always use environment variables or secret management tools.
  4. Build Your First Agent

    Edit agent/agent.py:

    from google.adk.agents import LlmAgent # Define your first agent assistant_agent = LlmAgent( # Unique identifier for logging and tracking name="personal_assistant", # The LLM brain (e.g., "gemini-2.0-flash", "gemini-2.5-pro") model="gemini-2.0-flash", # Short description for routing and delegation description="A helpful personal assistant.", # System instructions - the agent's "personality" instruction="""You are a helpful personal assistant. Be friendly, concise, and always ready to help.""" )
  5. Run Your Agent
    # Option 1: Interactive Web UI (best for debugging) adk web # Option 2: Command Line Interface adk run # Option 3: Programmatic (for production) python agent.py

🎉 Congratulations!

You've just created your first AI agent! But this agent can only chat. To make it truly useful, we need to give it tools to interact with the world.

🧠 Core Concepts of ADK

Understanding Agents

📖 Definition: What is an Agent?

An agent is an application that attempts to achieve a goal by observing the world and acting upon it using an LLM and the tools at its disposal.

The Developer's View: From API Call to Reasoning Loop

User Input
→
LLM Reasoning
→
Tool Selection
→
Tool Execution
→
Response

This loop continues until the agent achieves its goal or determines it cannot proceed.

Types of Agents in ADK

Agent Type Main Component Primary Function Best Use Case
LlmAgent LLM (e.g., Gemini) Reasoning & Acting Chatbots, routers, dynamic decision-making
WorkflowAgent Pre-defined pattern Orchestration Sequential/parallel/loop workflows
CustomAgent Your Python Code Custom Logic Complex conditional branching

🎭 Real-World Example: Travel Planning System

Imagine building a travel booking system:

  • LlmAgent: Main coordinator that understands user intent ("I want to visit Paris")
  • WorkflowAgent (Sequential): Orchestrates the booking flow: search flights → book flight → search hotels → book hotel
  • WorkflowAgent (Parallel): Searches flights and hotels simultaneously to save time
  • CustomAgent: Handles payment processing with specific business logic

The Anatomy of an LlmAgent

from google.adk.agents import LlmAgent assistant_agent = LlmAgent( # The unique identifier # Used for logging and inter-agent communication name="assistant_agent", # The "brain" - the LLM model model="gemini-2.0-flash", # Short summary for OTHER agents to read # Helps a router decide when to delegate description="A helper agent that can answer questions.", # The "soul" - system prompt defining persona and behavior instruction="""You are an agent to help answer questions. Be concise and helpful.""", # Tools the agent can use (we'll cover this next!) tools=[] )

🛠️ Tools: Giving Your Agent Superpowers

🎯 The Problem: Agents Need to Act, Not Just Talk

Imagine asking your assistant: "What time is it?"

Without tools, the agent can only say: "I'm sorry, I don't have access to the current time."

With tools, the agent can actually check the time and give you a real answer!

📖 What are Tools?

Tools are functions that transform an LLM from a conversationalist into a capable assistant. They enable agents to perform actions beyond text generation—like checking databases, calling APIs, running code, or searching the web.

Types of Tools in ADK

⚙️

Function Tools

Custom Python functions decorated with @tool

📦

Built-in Tools

Pre-configured tools like Google Search, Code Execution, BigQuery

🌐

OpenAPI Tools

Auto-generated from OpenAPI specifications

🔌

MCP Tools

Model Context Protocol tools for standardized integrations

🤝

Third-Party Tools

LangChain, CrewAI, and other framework tools

☁️

Google Cloud Tools

Calendar, Drive, Spanner, and 100+ connectors

Creating Your First Tool

from google.adk.tools import tool import datetime @tool def get_current_time(city: str) -> str: """Returns the current time in the specified city. Args: city (str): The name of the city Returns: str: Current time in that city """ # Implementation if city.lower() == "new york": tz = "America/New_York" else: tz = "UTC" now = datetime.datetime.now() return f"The current time in {city} is {now.strftime('%H:%M:%S')}"

✨ Key Elements of a Tool:

  • @tool decorator: Registers the function as a tool
  • Type hints: Help the LLM understand inputs/outputs
  • Docstring: Explains the tool's purpose to the LLM (crucial!)
  • Return value: Provides data back to the agent

Registering Tools with Your Agent

@tool def check_calendar(date: str) -> str: """Checks calendar for events on a specific date. Args: date: Date in YYYY-MM-DD format """ events = { "2025-09-03": "Team Meeting at 10 AM", "2025-09-05": "Project Deadline" } return events.get(date, "No events scheduled.") # Create agent with tools assistant = LlmAgent( name="personal_assistant", model="gemini-2.0-flash", instruction="You are a helpful assistant.", tools=[get_current_time, check_calendar] # Register tools here! ) # Now when user asks "What time is it?" or "Do I have meetings tomorrow?" # the agent will automatically use the appropriate tool!

  • Cannot be used in sub-agents

OpenAPI Tools: Instant API Integration

💡 The Challenge

Your Pet Store API has 50 endpoints. Do you really want to write 50 function tools by hand?

Solution: OpenAPI tools automatically generate tools from your API specification!

from google.adk.tools.openapi_tool.openapi_spec_parser import OpenAPIToolset # Load OpenAPI specification with open("petstore_api.json", "r") as f: openapi_spec = f.read() # Auto-generate tools from spec petstore_toolset = OpenAPIToolset( spec_str=openapi_spec, spec_str_type="json", auth_scheme="bearer", auth_credential="API_KEY_HERE" ) # Create agent with OpenAPI tools api_agent = LlmAgent( name="petstore_assistant", model="gemini-2.0-flash", tools=[petstore_toolset], instruction="Help users interact with the Pet Store API." )

Tool Context: Making Tools Stateful

🎯 What is ToolContext?

ToolContext is a special object passed to tools that provides access to session information, state, and contextual data. It transforms tools from stateless functions into intelligent, coordinated components.

from google.adk.tools import ToolContext @tool def update_user_preferences( theme: str, language: str, tool_context: ToolContext ) -> str: """Update user preferences in session state.""" # Write to state tool_context.state['user_theme'] = theme tool_context.state['user_language'] = language return f"Preferences updated: {theme}, {language}" @tool def get_personalized_greeting(tool_context: ToolContext) -> str: """Get greeting based on stored preferences.""" # Read from state lang = tool_context.state.get('user_language', 'en') greetings = {'en': 'Hello', 'es': 'Hola', 'fr': 'Bonjour'} return greetings.get(lang, 'Hello')
💡 Pro Tip: Always check if there's a built-in tool before building your own! ADK provides tools for Google Search, Workspace (Calendar, Drive, etc.), Spanner, and more.

🧠 Memory & State: Making Agents Remember

🐟 The "Goldfish" Challenge

Turn 1:

User: "Hi, my name is Alex. What's the weather in Seattle?"

Agent: "Hello Alex! The weather in Seattle is sunny, 72°F."

Turn 2:

User: "What's my name?"

Agent: "I'm sorry, I don't know your name."

😱 The agent forgot!

⚠️ The Problem: Agents Have No Memory by Default

Without memory, each conversation turn is isolated. The agent can't remember previous interactions, user preferences, or context from earlier in the conversation.

Sessions: The Conversation Thread

📖 What is a Session?

A session is the current conversation thread. It stores:

  • Events: The conversation history (user messages, agent responses, tool calls)
  • State: Short-term "scratchpad" for storing data during the conversation
  • Metadata: Session ID, user ID, app name
{ "id": "session_e4a1-b8d2", "appName": "weather_bot_app", "userId": "developer_123", "state": { "last_city_checked": "London", "temperature_unit": "Fahrenheit", "user_tier": "premium" }, "events": [ { "author": "user", "content": { "parts": [{"text": "What is the weather like in London?"}] } }, { "author": "assistant", "content": { "parts": [{"text": "The weather in London is..."}] } } ] }

Types of Session Services

Type How It Works Persistence Best For
InMemorySessionService Stores data in application memory None (lost on restart) Quick tests, local development
DatabaseSessionService Stores in PostgreSQL, MySQL, SQLite Yes (survives restarts) Self-managed persistence
VertexAiSessionService Uses Vertex AI infrastructure Yes (managed by Google Cloud) Production on Google Cloud

Using Session State

from google.adk.sessions import InMemorySessionService # Create session service session_service = InMemorySessionService() # Create a session session = await session_service.create_session( app_name="my_app", user_id="user_123" ) # Access and modify state session.state['user_name'] = 'Alex' session.state['preferences'] = {'theme': 'dark'} # State persists across turns in the same session!

Agentic Memory: Long-Term Context

🎯 What is Agentic Memory?

Agentic Memory goes beyond sessions—it provides long-term context that persists across multiple conversations. Think of it as the agent's "long-term memory."

Session vs Memory

Session: Current conversation (short-term)

Memory: Across all conversations (long-term)

Use Cases

  • User preferences
  • Past interactions
  • Learned patterns
  • Historical context
from google.adk.memory import PreloadMemoryTool # Create agent with memory agent = LlmAgent( name="stateful_agent", model="gemini-2.0-flash", instruction="""You are a Vehicle Voice Agent. Use past interactions to provide personalized responses.""", tools=[PreloadMemoryTool()] # Enables memory retrieval ) # Memory is automatically retrieved at the start of each turn # and included in the system instruction
💡 Pro Tip: By default, ADK uses in-memory storage locally and Vertex AI Memory Bank when deployed to production. This makes development seamless!

🤝 Multi-Agent Systems: Building Agent Teams

🎯 Real-World Scenario: Travel Planning

Imagine you're building a travel assistant. A single agent trying to do everything gets confused:

  • Should I search for flights or hotels first?
  • What about restaurant recommendations?
  • How do I handle payment processing?
  • What if something goes wrong?

Solution: Break it into specialized agents, each with a clear job!

The Multi-Agent Approach

Travel Planning System Architecture

🎯

Coordinator Agent

Main agent that understands user intent and delegates to specialists

✈️

Flight Agent

Specialist in flight searches and bookings

🏨

Hotel Agent

Expert in accommodation searches and reservations

🍽️

Sightseeing Agent

Provides local recommendations and activities

Agent-as-Tool Pattern

💡 Key Concept: Agents Can Be Tools!

In ADK, specialized agents can be wrapped as tools and used by other agents. This creates powerful hierarchical systems.

# Step 1: Create specialist agents flight_agent = LlmAgent( name="flight_agent", model="gemini-2.0-flash", description="Expert in finding and booking flights", instruction="Search for flights and provide options." ) hotel_agent = LlmAgent( name="hotel_agent", model="gemini-2.0-flash", description="Expert in finding and booking hotels", instruction="Search for hotels and provide options." ) # Step 2: Wrap agents as tools from google.adk.tools import AgentTool flight_tool = AgentTool(agent=flight_agent) hotel_tool = AgentTool(agent=hotel_agent) # Step 3: Create coordinator that uses these tools coordinator = LlmAgent( name="travel_coordinator", model="gemini-2.0-flash", instruction="Help users plan trips by delegating to specialists.", tools=[flight_tool, hotel_tool] ) # Now coordinator can intelligently delegate to specialists!

Workflow Agents: Orchestration Patterns

➡️

SequentialAgent

Executes agents one after another in a specific order

Use case: Step-by-step processes

⚡

ParallelAgent

Runs multiple agents simultaneously

Use case: Independent tasks that can run concurrently

🔄

LoopAgent

Repeats an agent until a condition is met

Use case: Iterative refinement

Advanced Example: Optimized Travel Workflow

from google.adk.agents import SequentialAgent, ParallelAgent # Step 1: Parallel search (flights + hotels simultaneously) parallel_search = ParallelAgent( name="parallel_search", agents=[flight_agent, hotel_agent] ) # Step 2: Sequential workflow travel_workflow = SequentialAgent( name="travel_workflow", agents=[ sightseeing_agent, # First: Get recommendations parallel_search, # Then: Search flights & hotels in parallel summary_agent # Finally: Create summary ] ) # Optimization achieved! 🎉 # Instead of: sightseeing → flights → hotels (3 sequential steps) # We have: sightseeing → (flights || hotels) → summary (faster!)

Quality Control: Review Loops

🎯 The Quality Challenge

Your travel summary agent creates a report, but how do you ensure it meets quality guidelines?

Solution: Add a reviewer agent that checks quality and triggers regeneration if needed!

# Create reviewer agent reviewer_agent = LlmAgent( name="summary_reviewer", model="gemini-2.0-flash", instruction="""Review the travel summary. Check for completeness and structure. Return 'pass' or 'fail' with feedback.""" ) # Add to workflow with loop from google.adk.agents import LoopAgent quality_loop = LoopAgent( name="quality_loop", agents=[summary_agent, reviewer_agent], max_iterations=3, # Try up to 3 times break_condition="pass" # Stop when review passes )
💡 Pro Tip: Multi-agent systems are not just about delegation—they're about creating specialized, maintainable, and scalable agent architectures. Each agent has a clear responsibility, making debugging and improvements easier.

🔌 MCP & A2A: Standardized Protocols

Model Context Protocol (MCP)

📖 What is MCP?

Model Context Protocol is an open protocol that standardizes how applications provide context to LLMs. Think of it as a universal adapter for connecting data sources to AI agents.

MCP Architecture

Your ADK Agent
(MCP Client)
→
MCP Protocol
→
MCP Server
(Database, API, etc.)

Pattern 1: ADK as MCP Client

Your ADK agent uses tools from existing MCP servers

from google.adk.tools.mcp_tool import MCPToolset mcp_tools = MCPToolset( server_url="http://localhost:8000" ) agent = LlmAgent( name="agent", tools=[mcp_tools] )

Pattern 2: ADK as MCP Server

Wrap your ADK tools in an MCP server to share with any MCP client

# Expose ADK tools via MCP from google.adk.mcp import create_mcp_server server = create_mcp_server( tools=[my_tool_1, my_tool_2] ) server.run()

Agent-to-Agent (A2A) Protocol

📖 What is A2A?

Agent-to-Agent Protocol is an open standard designed to enable communication and collaboration between AI agents, even across different systems and organizations.

💡 Use Case: Imagine your customer service agent (running on your infrastructure) needs to communicate with a payment processing agent (running on a partner's infrastructure). A2A makes this seamless!

⭐ Best Practices & Security

Security Best Practices

🔐

Credential Management

  • Never hardcode API keys
  • Use secret management tools
  • Rotate credentials regularly
  • Use service accounts in production
🛡️

Input Validation

  • Validate all user inputs
  • Sanitize data before processing
  • Implement rate limiting
  • Use type hints and validation
🚨

Error Handling

  • Implement try-catch in tools
  • Use exponential backoff
  • Provide meaningful error messages
  • Log errors for debugging
🔍

Monitoring & Logging

  • Track agent performance
  • Monitor tool execution
  • Log important events
  • Set up alerts for failures

Tool Error Handling

❌ Common Mistakes:

  • Silent failures: Tools that fail without clear error signals
  • Infinite retry loops: No retry limits or backoff
  • Missing timeouts: Operations that block indefinitely
  • Ignoring ToolContext: Missing valuable error management capabilities

✅ Best Practices:

@tool def api_call_with_error_handling(url: str) -> dict: """Safe API call with proper error handling.""" try: # Implement timeout response = requests.get(url, timeout=10) response.raise_for_status() return {"status": "success", "data": response.json()} except requests.Timeout: return { "status": "error", "error_type": "timeout", "error_message": "API request timed out", "suggestions": ["Try again later"] } except requests.RequestException as e: return { "status": "error", "error_type": "api_error", "error_message": str(e), "suggestions": ["Check API endpoint", "Verify credentials"] } except Exception as e: # Log unexpected errors logger.error(f"Unexpected error in api_call: {e}") return { "status": "error", "error_type": "unknown", "error_message": "An unexpected error occurred" }
